Hy for Fun and Profit

2020-05-08

Well maybe not profit…

Python is a language famous for being so similar to natural languages that it appears to be almost the same as pseudocode. This has resulted in it becoming tremendously popular and given it a massive ecosystem of different libraries that allow you to do all sorts of interesting things: from scientific computing to cluster automation Python does everything. Python also serves as a good language for beginners and thus, unfortunately, is often looked down on for not being a sufficiently 1337 programming language.

Lisp is a family of different languages that all use prefix notation, a syntax based on S-expressions, and have macros that allow the user to trivially extend the language with what amounts to their own DSL for better or for worse. Lisps also tend to be fairly alien in terms of syntax though they do as a whole have a fairly regular syntax making them easy to understand and use. They also are infamous for allowing the user to feel absolutely superior to any other programmer- though Haskell and Rust have begun to threaten it in this regard.

Hy is the progeny of both Lisp and Python and thus has traits of both. It works well for personal projects and I have used it extensively in my attempts to produce a more automated desktop environment for myself. From automatic invoice generation to my own taskwarrior extensions that alert me to upcoming tasks Hy has been extremely enjoyable.

The Good

Macros

The first thing that made me love Hy was it’s macros. There is something magical about being able to write macros to simplify work in Python. Really, macros make Python so much more fun to use as they allow for expanded introspection and the development of small DSL like language extensions that are more clear than the usual operator overloading in Python.

Consistent Syntax

The syntax of Hy, like all Lisps, is fairly consistent. Obviously there are some things such as quoting lists or other such shorthand that makes the syntax not completely regular, but it is far more consistent than in Python. The context free syntax also allows for structure editing and other nice advanced text editor features, something that the non context free Python cannot easily use.

A Decent REPL

The Hy REPL is not perfect, but it is far better than the Python REPL. Because you can execute code one s-expression at a time you can easily incrementally build your own programs and test out your program as you go along on a much more granular level than with Python.

The Bad

Startup Times

Our first problem will be that the startup time of the transpiler and interpreter stack can be rather excessive seconds. Consider the code below:

#!/usr/bin/env hy

(print "hello world")

The first time execution of the code took a total of 0.775 seconds. After that it took a total of 0.09 seconds to execute. The latter time is not as terrible as the initial execution time, but is still fairly bad. Contrast with python which takes 0.03 seconds to execute the same script and requires that time for every execution of the script.

Low Adoption

The second problem is the lack of good IDE features. From my perspective REPL makes up for this, but many people would obviously disagree and I cannot deny that the lack of autocompletion and other similar fetures is regrettable.
